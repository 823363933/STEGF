## 1. 设计目标与基本判断

我希望构建的，并不是一个“由网格直接生成高斯”的系统，而是一个真正意义上的欧拉—拉格朗日解耦框架。在这个框架里，网格与高斯粒子承担不同职责：网格负责描述空间中的状态场，高斯粒子负责承载可追踪的物质与可渲染几何。换句话说，我不把网格理解为高斯模板的存储器，而把它理解为一个层级稀疏、可约束、可反馈的欧拉状态场；相应地，高斯粒子也不再是随时间从 cell 中临时生成的局部变量，而是具有持续身份、可演化属性和完整生命周期的拉格朗日表示。

这样做的原因很直接。若网格只承担 `cell feature → decoder → temporary gaussian` 这一类职责，那么它更像训练过程中的中间工具，而不是最终系统中真正有物理和结构意义的表示。相反，如果我要建立一个能够支持动态场景、粒子生命周期控制、局部约束和内容自适应分辨率的统一模型，那么网格必须是“场”，粒子必须是“物”，两者之间还必须存在双向交互，而不只是单向的条件调制。原文这部分的核心判断是明确的：高斯应当持久存在，网格应当表达空间状态，并且粒子与网格需要形成 `grid → particle` 和 `particle → grid` 的闭环耦合

## 2. 理论基础：欧拉场与拉格朗日载体的分工

从理论上说，这一设计可以概括为一句话：欧拉网格表示“空间中的场”，拉格朗日高斯表示“在场中运动的物质”。这与流体、MPM、PIC/FLIP 一类方法在思想上是一致的，只不过这里的粒子不是普通数值粒子，而是具备渲染能力的 Gaussian primitives。也正因为如此，我希望系统最终既不是纯场表示，也不是纯粒子表示，而是两者的结合：由欧拉状态场提供局部动力学、约束与控制，由拉格朗日高斯承担显式几何和渲染实体。

在这个框架下，网格至少承担三类功能。第一，它必须能够对任意空间位置和时间提供局部状态，也就是给出一个随 (x,t) 变化的场 (\mathcal{F}(x,t))。第二，它必须能够对位于空间中的高斯粒子施加动力学作用，包括局部速度、局部梯度、占据支持以及控制信号。第三，它还必须接收来自粒子的统计反馈，例如密度、动量、活动强度和重建误差，从而使网格不只是一个被查询的背景，而是一个持续更新、带有内部状态的空间系统。原文在这里强调得很清楚：最终网格不应再存“高斯模板参数”，而应存“空间状态和约束”

## 3. 网格的结构形式：层级稀疏欧拉状态网格

基于上述分工，我采用的主表示是层级稀疏欧拉状态网格。这里“层级”意味着我保留 coarse-to-fine 的多尺度分辨率结构，以便在大范围上维持全局覆盖，同时在局部高复杂度区域提供更细的表达能力；“稀疏”意味着我不会在整个空间上密集展开所有分辨率，而只在真正有几何、动态或误差需求的区域激活更细层级；“欧拉状态网格”则强调它不是隐式哈希缓存，也不是单纯的特征表，而是一个具有明确空间结构和状态语义的场表示。

为了让这种网格真正具备“场”的性质，我认为它至少需要四个结构特征。首先是多层级结构，用以支持粗到细的状态表达。其次是稀疏激活机制，用以把计算与参数预算集中在有意义的区域。再次是显式邻域关系，也就是每个 cell 都要有明确的中心、尺度、层级、父子关系和邻接关系，因为后续无论是插值、局部约束还是粒子回写，都依赖这种结构化拓扑。最后是状态分槽，也就是每个 cell 内部不是一整块语义不清的 latent，而是按功能拆分成若干状态通道。原文对这几点的概括也很准确：多层级、稀疏激活、显式邻域和状态分槽共同保证了网格的欧拉结构意义、内容自适应能力和粒子耦合能力

## 4. Cell 内部应存储什么

我把每个 cell 中的信息划分为五类。

第一类是几何元数据。这部分不属于学习参数，而属于网格结构本身，包括 cell 的中心、步长、层级、父子关系和邻域连接。它们的作用是定义空间拓扑，支持插值、邻域计算、层级细化以及粒子查询和粒子回写。换言之，这部分更适合作为 layout metadata 或 buffer，而不是训练参数。

第二类是静态状态槽。这部分描述一个空间位置随时间变化较弱、或者长期稳定的先验信息，比如静态上下文、几何先验和材质先验。这里的静态上下文用于表示该位置的长期稳定特征，几何先验用于表达该处更可能是边界、表面还是空域，材质先验则用于表达该处长期的外观承载倾向。这些量大多应作为可学习参数存在，因为它们需要通过长期训练从数据中塑造出来。

第三类是动态状态槽。这部分对应真正随时间变化的欧拉场状态，例如动态上下文、局部速度状态、局部占据状态和局部形变状态。这里最关键的一点是：这些量并不是高斯属性本身，而是空间位置在时刻 (t) 的场状态。它们可以通过时间基函数、递推模块或者粒子反馈共同构成，但语义上始终属于“场”。

第四类是控制与约束槽。这类量不是用于直接渲染的，而是用于控制粒子系统和网格系统自身的演化，例如 birth、death、split、merge、refine、boundary 和 consistency 之类的分数。它们决定某一区域是否应当生成新粒子、淘汰粒子、增加粒子密度、合并简化，或者是否应当提升网格分辨率、施加更强约束。我认为这一部分非常关键，因为它决定了系统是否具备真正的自适应和生命周期管理能力。

第五类是统计反馈槽。这部分通常不是主参数，而是运行时状态，用于记录粒子密度、动量、活动强度、渲染残差、可见性和动态变化等统计量。它们不是“网络需要直接学习出来的 latent”，而是系统运行过程中逐步积累的证据，因此更适合用 EMA 形式作为 buffer 保存，再用于后续的细化、回收和控制决策。你原文对 cell 的五类状态其实拆得已经很完整了，包括静态槽、动态槽、控制槽和 EMA 统计槽，并且明确区分了哪些更像 Parameter，哪些更像 Buffer

## 5. Parameter、Buffer 与混合项

在实现上，我会明确区分哪些变量应作为 Parameter，哪些应作为 Buffer。静态上下文、几何先验、材质先验，以及部分动态状态的基础系数、占据参数化和控制分数底座，应当作为可学习参数，因为这些量需要通过图像监督和辅助损失共同训练。相反，几何布局本身以及各类统计 EMA，更适合放在运行时状态中，因为它们更接近系统证据而不是可泛化参数。

除此之外，我认为还应保留一类混合项，即一个状态既有可学习底座，又允许接受运行时反馈修正。占据量就是一个典型例子：它既可以来自可学习的 occupancy latent，也可以叠加由粒子 splat 回写得到的 occupancy feedback。这样的设计能够把“场的先验”和“粒子的事实”结合起来，使系统比纯参数场或纯统计场都更稳定。

## 6. 时间参数化方式

我会保留时间基函数建模，但会改变其解释方式。时间调制的目标不应是“直接生成当前时刻的高斯参数”，而应是“恢复当前时刻 cell 的欧拉状态”。也就是说，如果使用 Fourier 特征或类似时间基展开，那么生成出来的不是高斯 primitive，而是 dynamic context、velocity state、occupancy state 等场状态通道。这样一来，你现在已有的时间建模思路仍然成立，但它在语义上会更加清晰：时间基函数服务于场，而不是直接服务于粒子的实例化。原文在这点上其实已经给出了明确方向，即 (s_i(t)) 对应的是 cell 的状态槽位，而不是高斯属性

## 7. 网格与粒子的耦合方式

我把网格与粒子的耦合分成三个过程。

第一个过程是 field query。给定某个粒子在空间中的位置，我从多层网格中插值得到该位置的局部场状态，包括速度、占据、动态上下文、材质上下文和控制信号。随后，这些量用于更新粒子的状态：位置由局部速度推进，形状由局部速度梯度和形变状态调节，外观由材质上下文和动态上下文修正，生命周期则由控制信号决定。换句话说，粒子的演化不是自身孤立完成的，而是由它所处位置的欧拉场持续驱动。

第二个过程是 particle splat。粒子在运动和渲染之后，会把自身的统计信息回写到周围 cell 中，例如密度支持、动量、活动强度、残差责任和可见性证据。这一步非常重要，因为只有通过这种回写，欧拉网格才真正“知道”粒子系统在发生什么，从而形成双向闭环。

第三个过程是场内约束求解。网格内部还需要进行邻域平滑、时序一致性、传输相关正则以及层级一致性约束。也就是说，欧拉场并不是一个被动存储和查询的表，而是一个内部带有规律、需要被持续规整的空间系统。原文把这一部分分成 Field Query、Particle Splat 和 Constraint Solve 三个阶段，这个划分本身就是合理的

## 8. 为什么主表示必须是层级稀疏，而不是 dense 或 hash

我不考虑用纯 dense 网格作为主表示，原因很简单：参数量和计算量会迅速失控，尤其是在大场景和高分辨率条件下，绝大多数空间都会被浪费在低价值区域上。另一方面，我也不打算让纯 hash grid 承担主欧拉场的职责。hash 结构很适合用于高频残差、局部细节缓存或参数压缩，但它缺少显式邻接关系和稳定的拓扑结构，因此不适合直接承担粒子—网格耦合、梯度传播、统计回写和层级约束这些任务。

相比之下，层级稀疏结构同时具备三方面优势：它能够通过 coarse 层保证全空间可查询性；能够通过 fine 层只在关键区域增加预算；同时又保留了明确的空间结构，便于施加欧拉式运算与约束。因此，作为最终主干表示，coarse-to-fine 的稀疏结构化欧拉场是更合理的选择。你原文对这一点的论证也很完整：dense 太重，hash 太弱结构，而层级稀疏恰好兼顾覆盖、效率与结构性

## 9. 内容自适应与层级重构

在自适应分辨率方面，我不主张每一步都重建整个网格层级，那样代价太高，也容易引入训练不稳定。更可行的方式是周期性重建 active hierarchy。具体来说，我会依据每个 cell 的活动强度、粒子密度、渲染残差、动态变化程度以及 refine score 构造一个综合分数，再周期性地决定哪些 coarse cell 需要展开为 finer children，哪些长期低价值的 fine cell 可以被合并或回收。

这样的机制有两个好处。第一，它让网格真正跟随内容生长，把分辨率集中投放到有几何、动态或误差压力的区域。第二，它与粒子统计天然耦合，因为活动更强、密度更高或误差更大的位置，往往也正是粒子系统更需要局部表达力的地方。原文这里提出了基于 `activity_ema`、`particle_density_ema`、`residual_ema`、`dynamic_variation_ema` 和 `refine_score` 的细化/回收分数，这个设计方向是很顺的

## 10. 物理与几何约束

关于“符合物理力学约束”这一点，我的想法不是一开始就把完整物理方程硬塞进系统，而是先把一个可学习的、带约束的欧拉场建立起来。具体来说，我会从四类约束入手。

第一类是时间平滑，约束动态上下文、速度状态和控制信号不要发生无意义的高频抖动。第二类是邻域一致性，约束相邻 cell 的速度、占据和动态状态在局部上保持合理连续性。第三类是传输一致性，即让某些状态在速度场的作用下满足类似 advection 的关系，从而体现“场随流动传播”的特征。第四类是结构分离，也就是把 occupancy、velocity、appearance context 和 control score 放在不同 head 中学习，避免所有因素混在一个隐变量里，减少语义纠缠。除此之外，我还会加入粒子—场一致性约束，例如粒子密度长期很高的区域，其 occupancy 不应长期偏低；粒子频繁经过的区域，其 activity 应当得到正确累积。原文这部分提出的时间平滑、邻域平滑、传输一致性、约束分离和粒子—场一致性，正好构成了一个比较完整的第一阶段约束集合

## 11. 训练流程

在训练层面，我希望系统流程从“网格直接吐出临时高斯再渲染”的单向过程，转变为“场—粒子—渲染—反馈”的闭环。一个更合理的训练周期应当包括：先构建或更新当前激活的层级稀疏欧拉场；在给定时刻恢复对应的欧拉状态；然后让粒子查询场并更新自身的位置、形状、外观和生命周期；之后对粒子集合进行渲染；再根据渲染误差和辅助约束共同优化场参数、粒子参数和耦合模块；最后把粒子统计回写到网格 buffer 中，并周期性调整 active hierarchy。原文对训练流程的这个闭环定义是很完整的，而且它和前面的理论分工是一一对应的

## 12. 一个简化但可落地的规格

若把上述思想压缩为一个可实现的最小系统，那么每个 cell 至少应包含以下字段：几何元数据，包括中心、步长、层级和拓扑关系；静态可学习参数，包括 `static_context`、`geometry_prior` 和 `material_prior`；动态可学习参数，包括 `dynamic_coeff`、`velocity_coeff`、`occupancy_coeff` 和 `control_coeff`；运行时统计状态，包括 `particle_density_ema`、`particle_momentum_ema`、`activity_ema`、`render_residual_ema`、`visibility_ema` 和 `dynamic_variation_ema`；在此基础上，后续还可以加入辅助性的外观残差或细节缓存，但它们不应成为主干表示。原文在“最终版网格规格建议”里给出的就是这套简化字段表

## 13. 总结

归根到底，我希望把最终版欧拉网格定义为一个层级稀疏、结构化、可约束、可接收统计反馈的欧拉状态场。它负责表示空间位置的长期状态、瞬时动态、局部控制以及与粒子的交互；而高斯粒子则负责承载可追踪的物质和可渲染几何。于是，网格保存的是“场”，粒子保存的是“物”，渲染主要来自粒子，动力学和规则主要来自场，而系统运行过程中产生的统计反馈又会重新回到场中，构成完整闭环。
